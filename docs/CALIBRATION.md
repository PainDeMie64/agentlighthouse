# Calibration

Phase 1.5 calibrated AgentLighthouse away from "all files exist, therefore 100" scoring.

## Repositories Tested

- AgentLighthouse itself: local devtool repository.
- `examples/sample-good-project`: high-readiness API/devtool sample.
- `examples/sample-project`: intentionally imperfect API sample.
- `examples/sample-bad-project`: intentionally weak Node sample.
- Optional external repos under `.tmp/validation-repos/`:
  - `sindresorhus/is`
  - `modelcontextprotocol/typescript-sdk`
  - `expressjs/expressjs.com`

External repos are cloned into `.tmp/validation-repos/` and are not committed.
Sanitized external reports are generated under `validation/reports/external/` when those repos are present locally; that directory is ignored because JSON reports can contain third-party evidence snippets.

## Scores Observed

After calibration:

- AgentLighthouse: high score, medium confidence, not perfect.
- sample-good-project: high score, not perfect.
- sample-project: low score.
- sample-bad-project: very low score.
- External repos: mixed scores with medium-to-low confidence depending on how much agent-specific context, benchmarks, and machine-readable API/MCP surface was available.

The exact current scores are regenerated by:

```bash
pnpm validate:realworld
```

## Scoring Changes Made

- Added score confidence and numeric confidence score.
- Added coverage reporting.
- Added scoring caps for missing agent instructions, missing docs, unknown project type, missing task benchmarks, shallow key artifacts, and low coverage.
- Added profile support: `default`, `devtool`, `api`, `docs`, `library`, `internal`.
- Added shallow artifact findings that look for command blocks, verification steps, grounded commands, troubleshooting, task criteria, concrete links, and ownership notes.

## False Positives Found

- Sample OpenAPI files inside `examples/` caused AgentLighthouse itself to be detected as an API project. Root self-scan now ignores example OpenAPI files for project classification.
- README command extraction previously misread `pnpm --filter` as a package script. Option flags are ignored.
- The freshness rule flagged its own documented rule ID because `deprecated` appeared inside inline code in `docs/RULES.md`. Freshness checks now strip inline code before matching stale terms.

## False Negatives Found

- A project could score 100 with MCP readiness not evaluated.
- A project could score 100 with shallow agent files that contained the right keywords.
- A project could score 100 without confidence or coverage context.

## Remaining Weaknesses

- OpenAPI analysis is still mostly textual.
- MCP analysis does not parse tool schemas yet.
- Docs usefulness is still deterministic and heuristic.
- Command verification checks whether commands are discoverable, not whether they actually run.
- Profiles are intentionally minimal and do not yet support rule-level severity overrides.

## What To Test Next

- More API repos with real OpenAPI examples.
- Python, Rust, and Go repositories.
- Docs sites where Markdown source is generated from another format.
- Repos with intentionally stale agent instruction files.
- MCP servers with real tool descriptions and schemas.
